# Замер точности SmartDoc

Этот каталог нужен для оценки качества чекера на реальных работах. Зачем — у
проекта SMART-цель «точность ≥ 85%», и эту цифру нужно подкрепить
экспериментом, иначе на защите она ничем не обоснована.

## Что нужно подготовить

1. Папка `benchmark/files/` — сложите туда **реальные работы** (свои или
   одногруппников, желательно 20+ файлов: смесь `.py` и `.docx`).
2. Файл `benchmark/ground_truth.json` — эталонная разметка: какие ошибки
   реально есть в каждом файле. Образец — `ground_truth.example.json`.

## Как сделать разметку

Это самая трудоёмкая часть, но без неё никак. Для каждого файла:

1. Откройте его и **сами** найдите все ошибки (как сделал бы преподаватель).
2. Запишите их в JSON в формате:
   ```json
   "имя_файла.py": {
       "true_issues": [
           {"line": 5, "code": "E225"},
           {"line": 12, "code": "F401"}
       ]
   }
   ```
3. Для `.docx` достаточно указать только `code` без строки.

Полный список кодов смотрите в `app/checkers/code_checker.py` (для Python) и
в `app/checkers/docx_checker.py` + `docx_extras.py` (для документов).

## Как запустить

```bash
python benchmark/run_benchmark.py
```

В консоли увидите по каждому файлу:
- TP (true positive) — сервис нашёл реально существующую ошибку
- FP (false positive) — сервис «придумал» ошибку, которой нет
- FN (false negative) — сервис пропустил реальную ошибку

И итоговые метрики:
- **Точность (precision)** — `TP / (TP + FP)` — какая доля найденного — правда.
- **Полнота (recall)** — `TP / (TP + FN)` — какую долю реальных ошибок поймали.
- **F1-мера** — гармоническое среднее, сводный показатель.

Подробный JSON с результатами окажется в `results.json`.

## Совет для защиты

Лучше показывать **F1**, а не только precision — это честнее. Если у вас
получится, скажем, точность 90% и полнота 75%, F1 будет около 82% — и это
надо включить в презентацию. На вопрос «почему не 100%» нормальный ответ:
«нормоконтроль — задача с интерпретацией, и часть случаев пограничные».

## Если не дотягиваете до 85%

1. Посмотрите, что чаще даёт FP — возможно, чекер слишком строгий по этому
   правилу. Можно ослабить (например, повысить tolerance).
2. Посмотрите, что чаще даёт FN — каких проверок не хватает в коде.
3. Расширьте `RULE_DESCRIPTIONS` или добавьте новые проверки в `docx_extras.py`.

После каждого изменения — прогоняйте бенчмарк заново. Это нормальный
итеративный процесс.
